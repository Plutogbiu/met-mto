# v1.1 版本更新需求

## 1. 版本信息

| 项目 | 内容 |
| --- | --- |
| 当前版本 | v1.0 |
| 目标版本 | v1.1 |
| 状态 | 开发中 |
| 创建时间 | 2026-07-09 |

## 2. 迭代目标

v1.1 主要围绕工单数据留痕和后台工单管理体验进行小版本增强：

1. 工单流转记录中的操作人改为真实姓名，方便后台审查和历史追溯。
2. 工单新增独立维保内容字段，避免把维保类别混写到工单内容中。
3. 工单“关闭”调整为“作废”，作废时必须填写原因并保留流程记录。
4. 增加工单物理删除能力，仅默认分配给管理员。

## 3. 本次更新清单

| 序号 | 更新项 | 说明 |

| 1 | 流程记录操作人真实姓名 | `operatorName` 保存用户 `realName`，不再使用账号名 |
| 2 | 历史流程记录姓名修正 | 根据 `operator_id` 关联 `sys_user.real_name` 回填历史数据 |

## 4. 工单流转记录操作人

### 4.1 调整目标

工单创建、编辑、状态变更、完成工单、现场处理记录等流程记录中，`operatorName` 需要保存用户真实姓名 `realName`。

### 4.2 后端调整

1. 登录生成 JWT 时写入 `realName`。
2. 鉴权拦截器解析 token 后写入 `currentRealName`。
3. 工单相关接口写流程记录时使用 `currentRealName`。
4. 旧 token 没有 `realName` 时，根据 `currentUserId` 查询用户真实姓名兜底。

### 4.3 历史数据修正

已有数据库升级时执行 `migration.sql` 中 `V20260709__sync_work_order_record_operator_real_name` 版本段：

```sql
update work_order_record r
inner join sys_user u on u.id = r.operator_id
set r.operator_name = u.real_name
where u.real_name is not null
  and u.real_name <> ''
  and (r.operator_name is null or r.operator_name <> u.real_name);
```

## 5. 工单维保内容

### 5.1 字段说明

| 字段 | 说明 |
| --- | --- |
| `maintenanceContent` | 前后端接口字段 |
| `maintenance_content` | 数据库字段 |

现场工单和日常巡检工单都需要填写维保内容。

### 5.2 当前前端选项

| 显示名称 | 字段值 |
| --- | --- |
| 保内免费 | `warranty_free` |
| 保外收费 | `out_warranty_paid` |

后端暂不做固定枚举校验，保持普通字符串字段接收。后续如需调整维保内容选项，优先调整前端选项或改为字典配置。

### 5.3 数据库变更

`schema.sql` 的 `work_order` 表已包含：

```sql

```

已有数据库升级时执行 `migration.sql` 中 `V20260709__add_work_order_maintenance_content` 版本段：

```sql
alter table work_order add column maintenance_content varchar(500) null comment '维保内容';
```

## 6. 工单作废

### 6.1 业务规则

1. 状态值沿用 `closed`，页面文案显示为“已作废”。
2. 后台工单管理中不再显示“关闭”，改为“作废”。
3. 作废时必须填写作废原因。
4. 作废原因不单独增加主表字段，写入工单流程记录。
5. 已完成和已作废工单不能再次作废。
6. 通用状态变更接口不能直接把工单改为 `closed`，必须调用作废接口。

### 6.2 流程记录规则

```text
record_type = void
status_before = 原状态
status_after = closed
content = 作废原因
```

### 6.3 后端接口

```http
PUT /api/admin/work-orders/{id}/void?reason=作废原因
```

权限沿用工单状态权限：

```text
work-order:status
```

## 8. 数据库升级内容

已有 v1.0 数据库升级到 v1.1 时，需要执行 `migration.sql` 中 v1.1 段：

1. `V20260709__sync_work_order_record_operator_real_name`
2. `V20260709__add_work_order_maintenance_content`
3. `V20260709__add_work_order_delete_permission`

本次涉及 `data.sql`，新环境初始化时会默认创建 `work-order:delete` 权限并分配给管理员。

## 9. 权限影响

1. 新增权限码 `work-order:delete`。
2. 作废接口沿用 `work-order:status` 权限。
3. 删除接口使用 `work-order:delete` 权限。
4. `data.sql` 和 `migration.sql` 默认只给管理员角色分配删除权限。

## 10. 工单物理删除

### 10.1 后端接口

```http
DELETE /api/admin/work-orders/{id}
```

权限：

```text
work-order:delete
```

### 10.2 删除范围

物理删除会删除以下数据库记录：

1. 工单指派工程师：`work_order_engineer`
2. 工单流程记录：`work_order_record`
3. 工单附件记录：`file_attachment` 中 `biz_type = work_order`
4. 流程记录附件记录：`file_attachment` 中 `biz_type = work_order_record`
5. 工单主记录：`work_order`

当前仅删除数据库附件记录，不删除服务器磁盘上的实际文件。

### 10.3 删除限制

1. 已完成工单不能删除。
2. 删除前后台需要二次确认。
3. 前端删除按钮按 `work-order:delete` 权限显示。

## 11. 部署范围

本次修改涉及：

1. 后端：需要重新打包部署 `mto-server`。
2. 后台：需要重新打包部署 `mto-admin`。
3. App：涉及状态文案和处理记录文案，正式 App 包需要重新打包；H5 调试环境重新构建即可。
4. 数据库：老环境升级需要执行 v1.1 对应 migration SQL。

## 12. 验收标准

1. 新建工单和编辑工单时，维保内容可独立选择并保存。
2. 工单列表、详情再次打开后，维保内容可以正常回显。
3. 创建、编辑、状态变更、完成、现场处理等流程记录显示真实姓名。
4. 历史流程记录执行 migration 后，账号名可回填为真实姓名。
5. 后台工单管理中不再出现“关闭”按钮，改为“作废”。
6. 点击作废时必须填写作废原因，未填写不能提交。
7. 作废后工单状态显示为“已作废”。
8. 工单详情流程记录中显示“作废工单”和对应作废原因。
9. 已完成、已作废工单不能再次作废。
10. App 中 `closed` 状态显示为“已作废”，处理记录中的 `void` 显示为“作废工单”。
11. 管理员拥有 `work-order:delete` 权限后，可以看到删除按钮。
12. 点击删除时出现二次确认弹窗。
13. 确认删除后，工单列表不再显示该工单。
14. 已完成工单调用删除接口时返回业务错误。
