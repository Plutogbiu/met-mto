# 运维内部工单系统功能需求分析

## 1. 项目定位

本系统用于公司内部运维团队管理客户现场工单、巡检和维保留痕。当前服务对象主要是煤矿客户现场，但代码、表名和接口统一使用“客户 / customer”表达，不使用 `mine` 作为代码命名。

团队规模约 15 人，不需要高并发和复杂流程。v1.0 目标是把“没有工单系统、没有历史记录、没有图片证据、没有打卡定位、没有处理回执沉淀”的问题先闭环。

## 2. 使用角色

| 角色 | 使用端 | 场景 | 主要操作 |
| --- | --- | --- | --- |
| 管理员 | 后台管理端 | 系统维护、人员权限管理 | 维护人员、权限、客户、设备、工单 |
| 线上运维 | 后台管理端 | 接收客户反馈、创建工单、查看处理结果 | 创建和派发工单、查看现场处理资料 |
| 现场实施工程师 | 移动 App | 到客户现场处理硬件或巡检工作 | App 查看工单、导航、打卡、上传图片、完成工单 |

现场实施工程师主要使用 App，不登录后台管理端。后台权限采用“角色默认权限 + 用户权限覆盖”的方式，便于后续按人员灵活调整。

## 3. 当前后台模块

后台 v1.0 包含以下模块：

1. 登录模块：真实接口，JWT + Redis，会话过期时间 30 分钟。
2. 权限控制：权限定义、角色权限、用户权限覆盖，后台操作按权限校验。
3. 客户管理：客户现场档案、腾讯地图选址、坐标维护、客户详情二级页面。
4. 人员管理：账号、姓名、手机号、角色、状态、重置密码、权限分配。
5. 设备管理：独立设备档案，不绑定客户现场。
6. 工单管理：现场工单、日常巡检工单、多人指派、时间筛选、状态处理。
7. 工单详情：基础信息、指派人员、打卡位置、状态流程、处理记录、图片资料审查。
8. 附件管理：仅后端接口，不在后台菜单中单独出现。

## 4. 当前移动端模块

移动端采用内部 App 使用方式，不需要上架应用商店。当前基础页面和流程已经完成：

1. App 登录。
2. 工单列表：首页只展示未完成工单，方便现场工程师处理待办。
3. 我的工单：保留全部历史工单，支持按状态、日期范围和关键词查询。
4. 工单详情：展示客户、设备、处理人、预计到达时间、附件和操作入口。
5. 现场打卡：获取当前位置，记录经纬度和定位地址。
6. 附件上传：支持多图上传和已上传图片回显。
7. 完成工单：完成前弹窗确认，并校验是否已经打卡。
8. 我的页面：个人信息、我的工单、设置、退出登录。

## 5. 工单类型

| 类型 | 代码值 | 说明 |
| --- | --- | --- |
| 现场工单 | `onsite` | 线上运维判断需要到现场后创建，通常关联设备 |
| 日常巡检工单 | `inspection` | 巡检作为工单的一种存在，不单独做复杂模块 |

现场工单需要设备、预计到达时间、预估完成时间。巡检工单不需要设备和预估完成时间。

## 6. 工单核心规则

1. 新建工单默认状态为 `pending`，即待处理。
2. 工单不需要人工填写题目，系统根据客户、设备和类型生成标题快照。
3. 客户现场数量可能超过 200 个，后台创建工单时客户选择必须支持搜索。
4. 工单可同时指派多个现场工程师。
5. 多人派单是一张工单关联多个工程师，不是给每个人生成独立工单。
6. 当前逻辑为多人协同同一张工单，一人完成后整张工单变为已完成。
7. 现场工单选择设备时，设备来自独立设备档案，不绑定客户。
8. 现场实施工程师只能看到指派给自己的工单。
9. App 首页只展示未完成工单，历史工单在“我的工单”中查询。
10. 完成工单前必须已有打卡记录。
11. 工单详情后台只做查看和审查，不在后台上传现场图片。

## 7. 工单状态

| 状态 | 代码值 | 说明 |
| --- | --- | --- |
| 待处理 | `pending` | 工单已创建，现场人员未开始 |
| 处理中 | `processing` | 已开始处理或已到现场 |
| 已完成 | `completed` | 现场处理或巡检已提交完成 |
| 已关闭 | `closed` | 工单关闭，作为历史记录保存 |

当前不做审批流程、不做复杂 SLA、不做自动派单。

## 8. 客户管理需求

客户管理用于维护客户现场基础信息，是工单、打卡、导航和巡检历史的基础数据。

客户档案包含：

1. 客户名称。
2. 地址。
3. 联系人和联系电话。
4. 经度、纬度。
5. 定位地址。
6. 定位备注。
7. 状态和备注。

客户详情二级页面需要展示：

1. 客户基础信息。
2. 历史现场工单记录。
3. 历史巡检工单记录。
4. 可跳转到对应工单详情查看图片、流程和处理记录。

## 9. 设备管理需求

设备是独立档案，不属于某一个客户现场。

当前设备字段保持简单：

1. 设备名称。
2. 设备型号。
3. 设备编号 / 序列号。
4. 状态。

已删除安装位置和备注，避免第一版字段过重。

## 10. 附件和图片分类

附件模块只做后端接口，不在后台菜单中单独展示。后台在工单详情里查看图片资料，移动端负责现场上传。

统一上传接口参数：

1. `file`：上传文件。
2. `bizType`：业务类型，例如 `work_order`。
3. `bizId`：业务 ID，例如工单 ID。
4. `category`：附件分类。

现场工单图片分类：

| 分类 | 代码值 |
| --- | --- |
| 原故障图片 | `fault_before` |
| 处理后图片 | `fault_after` |
| 现场处理图片 | `site_process` |
| 工单处理回执 | `work_receipt` |

巡检工单图片分类：

| 分类 | 代码值 |
| --- | --- |
| 巡检图片 | `inspection_photo` |
| 巡检处理回执 | `inspection_receipt` |

## 11. 当前暂不做的功能

1. Docker Compose 部署编排。
2. MinIO 存储。
3. 复杂审批流。
4. 巡检清单模板管理。
5. 报告模板编辑器。
6. 高并发优化。
7. 大屏和复杂统计报表。
8. 实时定位轨迹。
9. 备件库存管理。
10. 打卡 3 公里范围限制。

## 12. 后续扩展方向

1. App 内部版本更新。
2. 打卡范围限制。
3. 巡检清单和巡检结果。
4. 回执报告生成。
5. 图片水印和压缩。
6. 文件存储切换到 MinIO 或对象存储。
7. 备份和部署脚本。
8. 数据库升级版本段持续维护。
