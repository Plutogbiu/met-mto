# 项目结构与界面原则

## 1. 项目结构

```text
met-mto
├── mto-admin    PC 管理后台
├── mto-server   后端 API 服务
├── mto-app      内部移动端
├── uploads      本地上传文件
└── docs         需求、架构、使用和开发约定文档
```

## 2. 目录职责

| 目录 | 职责 |
| --- | --- |
| `mto-admin` | 管理后台，供管理员和线上运维使用 |
| `mto-server` | 后端接口、数据库访问、鉴权、权限、文件上传、腾讯地图接口 |
| `mto-app` | 内部 App，供现场实施工程师使用 |
| `docs` | 功能需求、技术架构、结构原则和开发维护约定 |
| `uploads` | 本地文件上传目录 |

## 3. 后台当前页面

后台不做营销页和复杂首页，登录后直接进入第一个有权限的业务页面，通常是工单管理。

当前后台页面：

1. 登录页。
2. 工单管理。
3. 工单详情。
4. 客户管理。
5. 客户详情。
6. 设备管理。
7. 人员管理。
8. 用户权限分配弹窗。

附件管理不单独出现在后台菜单中，只在工单详情中按分类查看图片资料。

## 4. App 当前页面

当前 App 页面：

1. 登录页。
2. 工单列表：只展示未完成工单。
3. 我的工单：查询全部历史工单。
4. 工单详情。
5. 现场打卡。
6. 附件上传。
7. 处理记录查看。
8. 我的页面。

底部导航保留“工单”和“我的”，打卡不作为底部导航入口。

## 5. 后台界面原则

1. 简约、清晰、适合反复使用。
2. 表格页优先支持搜索、筛选、分页和快速进入详情。
3. 表单字段不一次性堆太多，后续根据实际使用补充。
4. 客户、工单、设备等选择项要支持搜索，避免 200 多个客户全量下拉导致难用。
5. 工单详情后台定位为审查和记录查看，不承担现场上传。
6. 图片以分组预览为主，支持点击放大查看。
7. 页面保持 8px 左右圆角，避免过度装饰。
8. 后台以工作效率为主，不做复杂动效。
9. 权限分配使用树形勾选，按模块分组展示功能点。

## 6. App 界面原则

移动端面向现场实施工程师，优先保证现场易用：

1. 采用简约、清爽、接近 Apple Design 的视觉方向。
2. 颜色与后台保持接近，以蓝色为主色，但避免过度单调。
3. 顶部内容避开手机状态栏，避免被时间、电池遮挡。
4. 首页只展示未完成工单，减少现场查找成本。
5. 我的工单保留历史查询能力，支持日期范围筛选。
6. 按钮大，操作少，重要操作必须二次确认。
7. 完成工单前校验是否已经打卡，避免误完成。
8. 上传图片支持多图和已上传回显。
9. 弱网环境下后续可考虑草稿和重试。

## 7. 客户详情页面原则

客户详情页用于快速回顾某个客户现场的历史情况。

页面内容：

1. 客户基础信息。
2. 客户坐标和定位备注。
3. 历史现场工单记录。
4. 历史巡检工单记录。
5. 工单详情跳转入口。

## 8. 工单详情页面原则

工单详情页后台用于审查，App 用于现场处理。

后台详情页内容：

1. 工单基础信息。
2. 指派工程师。
3. 打卡位置。
4. 处理记录和状态流程。
5. 图片资料审查。

App 详情页内容：

1. 工单基础信息。
2. 客户和设备信息。
3. 预计到达时间和预估完成时间。
4. 附件分组查看。
5. 现场打卡入口。
6. 附件上传入口。
7. 完成工单按钮。

图片资料按工单类型分组：

1. 现场工单：原故障图片、处理后图片、现场处理图片、工单处理回执。
2. 巡检工单：巡检图片、巡检处理回执。

## 9. 数据库脚本维护原则

数据库脚本按职责拆分：

1. `schema.sql`：只放当前完整建表结构。
2. `data.sql`：只放初始化账号、权限和角色权限。
3. `migration.sql`：只放已有数据库版本升级补丁。

后续新增字段时，不只修改 `schema.sql`，还要在 `migration.sql` 追加版本段，方便服务器已有数据库升级。

版本段命名建议：

```sql
-- V20260708__add_xxx_field
-- 说明：本次变更目的。
```

## 10. 后续页面边界

短期不再扩张复杂后台页面，优先补强：

1. App 内部更新。
2. 打卡范围限制。
3. 回执报告生成和查看。
4. 图片压缩、水印和上传稳定性。
5. 巡检清单模板。
