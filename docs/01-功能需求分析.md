# 运维内部工单系统功能需求分析

## 1. 项目定位

本系统用于公司内部运维团队管理客户现场工单、巡检和维保留痕。当前服务对象主要是煤矿客户现场，但代码、表名和接口统一使用“客户 / customer”表达，不使用 `mine` 作为代码命名。

团队规模约 15 人，不需要高并发和复杂流程。第一阶段目标是把“没有工单系统、没有历史记录、没有图片证据、没有回执报告”的问题先闭环。

## 2. 使用角色

| 角色 | 场景 | 主要操作 |
| --- | --- | --- |
| 管理员 / 线上运维 | 接收客户反馈、创建工单、查看处理结果 | 登录后台、维护客户/人员/设备、创建和跟踪工单 |
| 现场实施工程师 | 到客户现场处理硬件或巡检工作 | App 查看工单、导航、打卡、上传图片、填写处理记录 |

当前不做复杂权限矩阵，先通过用户角色区分管理员、线上运维、现场工程师。后续人员规模扩大后再增加细粒度权限。

## 3. 当前后台模块

后台基础模型暂时完成，包含以下模块：

1. 登录模块：真实接口，JWT + Redis，会话过期时间 30 分钟。
2. 客户管理：客户现场档案、腾讯地图选址、坐标维护、客户详情二级页面。
3. 人员管理：账号、姓名、手机号、角色、状态、重置密码。
4. 设备管理：独立设备档案，不绑定客户现场。
5. 工单管理：现场工单、日常巡检工单、多人指派、状态维护。
6. 工单详情：基础信息、指派人员、状态流程、处理记录、图片资料审查。
7. 附件管理：仅后端接口，不在后台菜单中单独出现。

## 4. 工单类型

| 类型 | 代码值 | 说明 |
| --- | --- | --- |
| 现场工单 | `onsite` | 线上运维判断需要到现场后创建，通常关联设备 |
| 日常巡检工单 | `inspection` | 巡检作为工单的一种存在，不单独做复杂模块 |

现场工单需要设备、预计到达时间、预估完成时间。巡检工单不需要设备和预估完成时间。

## 5. 工单核心规则

1. 新建工单默认状态为 `pending`，即待处理。
2. 工单不需要人工填写题目，系统根据客户、设备和类型生成标题快照。
3. 客户现场数量可能超过 200 个，后台创建工单时客户选择必须支持搜索。
4. 工单可同时指派多个现场工程师。
5. 现场工单选择设备时，设备来自独立设备档案，不绑定客户。
6. 工单详情后台只做查看和审查，不在后台上传现场图片。

## 6. 工单状态

| 状态 | 代码值 | 说明 |
| --- | --- | --- |
| 待处理 | `pending` | 工单已创建，现场人员未开始 |
| 处理中 | `processing` | 已开始处理或已到现场 |
| 已完成 | `completed` | 现场处理或巡检已提交完成 |
| 已关闭 | `closed` | 工单关闭，作为历史记录保存 |

当前不做审批流程、不做复杂 SLA、不做自动派单。

## 7. 客户管理需求

客户管理用于维护客户现场基础信息，是工单、打卡、导航和巡检历史的基础数据。

客户档案包含：

1. 客户名称。
2. 地址。
3. 联系人和联系电话。
4. 经度、纬度。
5. 定位地址。
6. 定位备注。
7. 状态和备注。

客户详情二级页面需要展示：

1. 客户基础信息。
2. 历史现场工单记录。
3. 历史巡检工单记录。
4. 可跳转到对应工单详情查看图片、流程和处理记录。

## 8. 设备管理需求

设备是独立档案，不属于某一个客户现场。

当前设备字段保持简单：

1. 设备名称。
2. 设备型号。
3. 设备编号 / 序列号。
4. 状态。

已删除安装位置和备注，避免第一版字段过重。

## 9. 附件和图片分类

附件模块只做后端接口，不在后台菜单中单独展示。后台在工单详情里查看图片资料，移动端后续负责现场上传。

统一上传接口参数：

1. `file`：上传文件。
2. `bizType`：业务类型，例如 `work_order`。
3. `bizId`：业务 ID，例如工单 ID。
4. `category`：附件分类。

现场工单图片分类：

| 分类 | 代码值 |
| --- | --- |
| 原故障图片 | `fault_before` |
| 处理后图片 | `fault_after` |
| 现场处理图片 | `site_process` |
| 工单处理回执 | `work_receipt` |

巡检工单图片分类：

| 分类 | 代码值 |
| --- | --- |
| 巡检图片 | `inspection_photo` |
| 巡检处理回执 | `inspection_receipt` |
