# 运维内部工单系统技术架构设计

## 1. 架构定位

本项目面向小团队内部使用，由单人开发和维护，采用轻量级模块化单体架构。当前不引入微服务、消息队列和复杂部署编排。

核心原则：

1. 后台、后端、移动端分目录维护，职责清晰。
2. 一套 Spring Boot 后端同时服务后台和 App。
3. 本地文件存储先跑通，后续上服务器后再按需要切换 MinIO。
4. MySQL 作为主数据库。
5. 腾讯地图用于客户选址、工单导航和打卡定位。
6. 数据库 SQL 按建表、初始化、版本补丁拆分，方便人工迭代。

## 5. 当前后端基础能力

1. 统一返回：`ApiResult`。
2. 分页返回：`PageResult`。
3. 全局异常：`exception` 包，包含异常枚举 `ErrorCode`。
4. 跨域配置。
5. MyBatis Plus 分页插件。
6. MyBatis Plus 控制台 SQL 日志。
7. JWT + Redis 登录鉴权。
8. 权限注解：`@RequirePermission`。
9. 角色默认权限 + 用户权限覆盖。
10. 现场实施工程师工单访问限制。
11. 本地上传文件静态访问：`/uploads/**`。
12. Jackson 日期格式化输出。

## 7. 数据库脚本约定

当前不引入 Flyway，采用人工执行 SQL 的轻量方式。

数据库脚本位于：

```text
mto-server/src/main/resources/db
├── schema.sql      只放建表语句
├── data.sql        默认账号、默认权限、角色权限
├── migration.sql   版本迭代字段补丁和历史数据修正
└── README.md       SQL 使用说明
```

新环境初始化：

```sql
source schema.sql;
source data.sql;
```

已有环境升级：

```text
打开 migration.sql
找到本次版本段
只执行对应 SQL
```

后续新增字段时，同时维护：

1. `schema.sql`：让新环境直接拥有最新完整表结构。
2. `migration.sql`：让旧环境可以按版本补字段或修数据。

## 8. 当前接口约定

后台接口统一使用 `/api/admin` 前缀。App 登录使用 `/api/app/auth`，工单和附件接口当前复用后台路径并通过角色、权限和指派关系限制访问。

| 模块 | 接口 |
| --- | --- |
| 后台登录 | `POST /api/admin/auth/login` |
| App 登录 | `POST /api/app/auth/login` |
| 客户管理 | `/api/admin/customer-sites` |
| 人员管理 | `/api/admin/users` |
| 用户权限 | `/api/admin/users/{id}/permissions` |
| 设备管理 | `/api/admin/devices` |
| 工单管理 | `/api/admin/work-orders` |
| 工单完成 | `PUT /api/admin/work-orders/{id}/complete` |
| 工单记录 | `/api/admin/work-orders/{workOrderId}/records` |
| 附件上传和查询 | `/api/admin/attachments` |
| 腾讯地图配置与搜索 | `/api/admin/maps/*` |

工单分页已支持：

1. `keyword`
2. `type`
3. `status`
4. `customerKeyword`
5. `unfinishedOnly`
6. `customerSiteId`
7. `engineerId`
8. `createdStart`
9. `createdEnd`
10. `page`
11. `size`

客户详情页直接复用 `customerSiteId + type` 查询历史现场工单和巡检工单。
现场实施工程师请求工单列表时，后端会自动按当前登录用户过滤指派工单。

## 11. 2G 轻量服务器建议

2G 服务器可以用于内部试运行，但需要控制资源：

1. 后端 JVM 设置最大堆内存，例如 512MB。
2. MySQL 使用轻量配置。
3. Redis 保持小内存使用。
4. 暂不部署 MinIO。
5. 开启 swap，降低偶发内存不足风险。
6. 图片在 App 端上传前尽量压缩。

如果后续图片和报告数量明显增加，建议升级到 4G 或更高配置。
