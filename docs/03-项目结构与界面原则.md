# 项目结构与界面原则

## 1. 项目结构

```text
met-mto
├── mto-admin    PC 管理后台
├── mto-server   后端 API 服务
├── mto-app      内部移动端
└── docs         需求和架构文档
```

## 2. 目录职责

| 目录 | 职责 |
| --- | --- |
| `mto-admin` | 管理后台，供线上运维和管理员使用 |
| `mto-server` | 后端接口、数据库访问、鉴权、文件上传、腾讯地图接口 |
| `mto-app` | 内部 App，供现场实施工程师使用 |
| `docs` | 功能需求、技术架构、结构和界面原则 |

## 3. 后台当前页面

后台不做营销页和复杂首页，登录后直接进入工单管理。

当前后台页面：

1. 登录页。
2. 工单管理。
3. 工单详情。
4. 客户管理。
5. 客户详情。
6. 设备管理。
7. 人员管理。

附件管理不单独出现在后台菜单中，只在工单详情中按分类查看图片资料。

## 4. 后台界面原则

1. 简约、清晰、适合反复使用。
2. 表格页优先支持搜索、筛选、分页和快速进入详情。
3. 表单字段不一次性堆太多，后续根据实际使用补充。
4. 客户、工单、设备等选择项要支持搜索，避免 200 多个客户全量下拉导致难用。
5. 工单详情后台定位为审查和记录查看，不承担现场上传。
6. 图片以分组预览为主，支持点击放大查看。
7. 页面保持 8px 左右圆角，避免过度装饰。
8. 后台以工作效率为主，不做复杂动效。

## 5. 客户详情页面原则

客户详情页用于快速回顾某个客户现场的历史情况。

页面内容：

1. 客户基础信息。
2. 客户坐标和定位备注。
3. 历史现场工单记录。
4. 历史巡检工单记录。
5. 工单详情跳转入口。

## 6. 工单详情页面原则

工单详情页用于后台审查，不用于现场上传。

页面内容：

1. 工单基础信息。
2. 指派工程师。
3. 处理记录和状态流程。
4. 图片资料审查。

图片资料按工单类型分组：

1. 现场工单：原故障图片、处理后图片、现场处理图片、工单处理回执。
2. 巡检工单：巡检图片、巡检处理回执。

## 7. 移动端界面原则

移动端后续面向现场实施工程师，优先保证现场易用：

1. 按钮大，操作少。
2. 工单信息一屏看重点。
3. 打卡、导航、上传图片要直观。
4. 图片分类不要让现场人员反复理解。
5. 弱网环境下尽量保留草稿。
6. 上传前压缩图片，减少服务器压力。

## 8. 后续页面边界

后台模型暂时完成后，短期不继续增加后台页面。后续优先开发：

1. App 登录。
2. App 工单列表和详情。
3. 到场打卡。
4. 现场图片上传。
5. 工单和巡检提交。
6. 回执报告生成和查看。
