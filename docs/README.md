# 运维内部工单系统设计文档

本目录记录 MET MTO 运维内部工单系统的当前产品边界、技术架构、项目结构和开发维护约定。

当前项目定位为小团队内部使用，主要解决客户现场工单、现场维保记录、巡检记录、图片留痕、打卡定位和后续报告沉淀问题。项目当前按 `v1.0.2` 阶段维护，后台管理端、后端基础接口、权限控制、移动端基础现场能力和完成工单前置校验已经完成。

## 当前状态

v1.0.2 当前已经完成：

1. 后台登录与 JWT/Redis 会话。
2. 后台权限控制：角色默认权限 + 用户单独权限覆盖。
3. 客户管理和客户详情二级页面。
4. 人员管理、权限分配、重置密码。
5. 独立设备管理。
6. 工单管理：现场工单、巡检工单、多人指派、时间筛选。
7. 工单详情：基础信息、指派人员、打卡位置、流程记录、图片资料审查。
8. 文件上传和附件分类接口。
9. 移动 App：登录、未完成工单、历史工单、工单详情、现场打卡、附件上传、完成工单、我的页面。
10. 数据库脚本拆分：`schema.sql`、`data.sql`、`migration.sql`。
11. 完成工单前置校验：至少 1 张工单附件，且所有指派工程师均已完成现场打卡。
12. 登录会话区分客户端：后台管理端 30 分钟，现场 App 7 天。

## 当前约定

1. 代码和数据库命名统一使用 `customer` 表达客户现场，不使用 `mine`。
2. 后端包结构采用顶层分层，`service.impl` 单独放实现类。
3. 数据库初始化和升级不引入 Flyway，当前通过人工执行 SQL 文件维护。
4. 新环境执行 `schema.sql` + `data.sql`。
5. 老环境版本升级执行 `migration.sql` 中对应版本段。
6. 现场实施工程师主要使用 App，不登录后台。
7. App 首页只展示未完成工单，历史工单从“我的工单”查询。

## 后续建议

短期优先级：

1. App 内部版本更新机制。
2. 工单回执报告生成。
3. 图片压缩、水印和上传体验优化。
4. 数据库备份和服务器部署脚本。
5. 巡检清单模板和巡检结果结构化。
